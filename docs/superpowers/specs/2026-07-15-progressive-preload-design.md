# Progressive Preload Design

## Goal

让 PC 和移动端在预加载结束时首屏已经可以看到并操作，同时避免未来新增的非首屏图片无限延长首次进入时间。

## Behavior

- 预加载层只等待首屏关键资源、首屏图片完成下载和解码、字体可用，以及页面基础启动任务完成。
- 只有以上条件满足后，进度才显示 `LOADING 100%` 并隐藏预加载层。
- 非首屏资源在预加载期间或进入页面后进入后台队列，不阻塞首次进入。
- 距离当前视口较近的图片优先于远处资源加载，保证用户快速滚动时内容及时出现。
- 单个资源失败不会阻塞页面；页面保留占位或现有备用显示，并记录失败资源供调试。
- PC 和移动端使用各自的首屏资源清单，移动端不预加载桌面端不需要的重型资源。

## Implementation Shape

1. 将资源分为 `critical`、`nearViewport` 和 `deferred` 三层。
2. `critical` 资源通过预加载器加载，并在图片 `decode()` 完成后计为 ready。
3. 首屏静态 DOM 和基础模块通过统一的 boot-ready 信号通知预加载器。
4. 页面进入后使用现有的懒加载机制和空闲任务加载其余资源；不把所有资源数量纳入首屏百分比。
5. 保留最大等待时间作为异常兜底，但兜底只用于避免永久遮罩，不将异常误报为正常资源完成。

## Success Criteria

- PC 和移动端首次进入时，首屏不出现资源仍在补齐的空白或明显跳变。
- `100%` 出现时，首屏核心图片已经完成解码，导航和基础交互已经挂载。
- 新增非首屏图片不会自动增加首次进入的阻塞时间。
- 离线或单个资源失败时，页面仍能进入并保持可用。
- `npm run build` 通过，且本地页面在 `127.0.0.1:5174` 返回成功。
